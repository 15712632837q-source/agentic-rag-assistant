# 工程决策记录

逐条说明"为什么这么选"——技术选型不是默认，而是有取舍的判断。

## 1. 用 LlamaIndex，而不是自研检索/agent 循环
我此前在另一个项目里**手写过**完整的 agent 壳（ReAct 循环、工具调度、MCP 客户端/服务端），所以不是不会造轮子。但这个项目刻意用 LlamaIndex：
- RAG 的脏活（切块、向量库抽象、检索器、持久化、引用回溯）是已被解决的问题，自研只会重复造轮子、增加维护成本；
- 团队里更看重"能快速上手既有生态"而非"什么都从零写"；
- 把精力留给真正有区分度的部分：agentic 检索策略、评测指标、生产化。

> 一句话：**会用生态快速交付 + 必要时能下沉手写**，两者都具备才是想要的工程画像。

## 2. 用 ReActAgent（agentic），而不是固定 RAG 管线
朴素 RAG（每问必检索一次再生成）对"不需要查/需要多轮查"的问题都不优。ReAct agent 让模型自主决定是否检索、检索什么、是否追问，更贴近真实助手行为，也更能体现 agent 编排能力。代价是多一两次 LLM 调用（延迟/成本），用指标量化权衡。

## 3. 生成用豆包/方舟，且做成可换引擎
- 通过 OpenAI 兼容接口（`OpenAILike`）接入，**换模型只改 `.env`**，不锁死单一厂商；
- `is_function_calling_model=False`：走文本式 ReAct，不依赖端点的原生 function calling，兼容性更稳；
- 必须**绕系统代理**（清代理环境变量直连）——本机系统代理会让方舟请求失败，这是踩过的坑。

## 4. embedding 用本地 bge-zh（fastembed），而非云端 embedding API
- 离线、零额外费用、不必再开通方舟 embedding 端点；
- `bge-small-zh` 对中文检索质量好、体积小（ONNX，无需 torch）；
- 代价：首次下载模型（约百 MB）。语料若暴增到百万级再换专业向量库（Chroma/PGVector）。

## 5. 先 CLI 跑通 → 再 FastAPI → 再 Docker
分层验证：先证明"检索+作答"正确，再包成服务，再容器化部署出真实可访问 URL。
"本地能跑"和"上线了"是两件事，作品集要的是后者。

## 6. 长期记忆：复用 RAG 的 embedding，而非另起一套
参考《从零搭建 AI-Agent 的 10 个模块》，agent 该有记忆。这里没引入向量数据库或额外模型：
- **存**：SQLite 一张表（text / kind / 时间戳 / 向量），轻、零依赖、可持久化；
- **取**：复用项目已配置的 bge embedding 算余弦相似度，再乘一个**指数时间衰减**
  （半衰期 14 天，旧记忆语义分被压低）——即"语义相关性 + 时间衰减"；
- **接法**：暴露 `remember` / `recall` 两个工具，由 ReActAgent **自主决定**何时存取，
  与 `knowledge_base` 检索工具并列。本质上"记忆"就是"对一个会增长的事实库做 RAG"，
  所以能直接复用现成基建。
- **边界**：SQLite 文件本地持久；Streamlit Cloud 文件系统易失，记忆在重启前有效（demo 够用）。

## 7. 技能（Skills）：markdown 手册 + 按需加载，不写死在 prompt 里
参考《从零搭建 AI-Agent 的 10 个模块》的 Skills。让 agent 能按【固定流程/格式】做事
（如概念对比、学习计划、要点速记），而不是每次自由发挥：
- **技能 = `skills/` 下的 markdown**（frontmatter 写 name/description，正文写流程/模板）。
  **新增技能 = 加一个 .md，零改代码**——用户可自建、也可装"社区技能"。
- **按需加载省 token**：`list_skills` 只把所有技能的【名称+描述】给模型；模型判断匹配后，
  再用 `load_skill(名称)` 取完整流程。技能很多时也不会撑爆上下文。
- **接法**：`list_skills` / `load_skill` 两个 FunctionTool，与 knowledge_base / 记忆工具并列，
  由 ReActAgent 自主调用。技能正文里要求"用 knowledge_base 取内容 + 按模板输出"，
  于是技能、检索、作答串成一条：加载技能→检索→按格式答（实测概念对比技能输出严格套模板）。
- **为什么不做成代码里的"模式"**：写成 markdown 才能让非程序员改/加技能，也对应笔记里的
  "skill creator + 社区技能"理念。
